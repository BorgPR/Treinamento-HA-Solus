Documentação: Arquitetura Solus Enterprise

Esta documentação descreve as decisões de engenharia aplicadas na infraestrutura de Alta Disponibilidade (HA) do Solus, detalhando como o sistema garante resiliência e continuidade em ambientes críticos.

Nota sobre Modularidade: Este modelo usa como referência um ambiente Local (On-Premise), mas sua estrutura é modular. Ele pode ser adaptado para Servidores em Nuvem (VPS) apenas alterando o componente de entrada de tráfego.

1. Camada de Aplicação (Cluster de Servidores)

Docker Swarm Proxy Nginx Segurança SSL

A aplicação roda em um cluster orquestrado. Cada servidor (nó) possui a cópia exata de todos os serviços, garantindo que se um hardware falhar, os outros assumam o trabalho sem interrupção.

  • Proxy Reverso: O Nginx dentro de cada servidor recebe as conexões e cuida da segurança SSL. Isso acelera o acesso e protege os sistemas internos (PHP e APIs) de acessos diretos.
  • Redistribuição de Carga: Em caso de queda, o Swarm move as réplicas para os nós ativos. Importante: Ao retornar, os serviços NÃO voltam sozinhos para o nó recuperado. Isso evita "efeito sanfona" e garante que o reequilíbrio ocorra apenas com autorização humana em janela de manutenção.

2. Camada de Armazenamento (Disco Compartilhado)

GlusterFS Replicação Tripla Auto-Correção

Em vez de um servidor de arquivos central (que seria um ponto único de falha), usamos armazenamento distribuído onde os discos de todos os servidores do cluster trabalham juntos.

  • Sincronismo: Os arquivos (PDFs, fotos, documentos) são gravados simultaneamente em 3 locais diferentes. Se um servidor queimar, nenhum dado é perdido.
  • Recuperação Automática: Quando um servidor volta a ficar online após uma falha, o sistema percebe os dados desatualizados e inicia a correção automática entre os discos.

3. Isolamento do Banco de Dados (Oracle)

Oracle 19c Hardware Dedicado

O Banco de Dados Oracle é mantido em servidores dedicados. Isso isola o "coração" dos dados da carga de processamento do site, garantindo estabilidade e evitando que uma lentidão no portal web afete a gravação das transações do banco.

4. Adaptação: Ambiente Local vs. Nuvem (VPS)

IP Virtual Balanceador de Carga Flexibilidade

O projeto foi desenhado para ser flexível. Dependendo de onde ele é instalado, mudamos apenas a porta de entrada:

  • Ambiente Local (Físico): Usamos a técnica de IP Virtual (VIP) com Keepalived. O IP "pula" de um servidor para o outro instantaneamente em caso de queda.
  • Ambiente em Nuvem (VPS): Como provedores de nuvem costumam bloquear IPs Virtuais manuais, substituímos o Keepalived por um Balanceador de Carga do próprio provedor (Cloud Load Balancer). O resto da estrutura (Nginx, Swarm e Discos) permanece exatamente igual.

5. Automação vs. Intervenção Manual

Recurso/Ação Tipo Descrição
Failover de IP (VIP) AUTOMÁTICO Migração do IP de entrada entre nós via Keepalived.
Redistribuição de Réplicas AUTOMÁTICO Na ocorrência de falha, os serviços migram para os nós ativos.
Rebalanceamento de Carga OPCIONAL Decisão estratégica entre manter a estabilidade atual ou forçar o retorno para o nó recuperado.
Sincronismo de Arquivos AUTOMÁTICO O GlusterFS mantém os blocos de dados idênticos em rede.
Auto-Healing (Storage) AUTOMÁTICO Recuperação de dados defasados quando um nó retorna ao cluster.
Troca de Hardware Físico MANUAL Substituição de peças, discos físicos ou cabos de rede.
Ajuste de Configuração MANUAL Alteração de regras de negócio ou limites de recursos da stack.
