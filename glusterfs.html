DocumentaÃ§Ã£o: Fundamentos da Alta Disponibilidade

Este apêndice detalha as decisões de arquitetura e o funcionamento técnico das tecnologias apresentadas no simulador.

Objetivo: Garantir que o ambiente Solus opere com tolerância a falhas tanto no processamento (Docker) quanto no armazenamento (GlusterFS).

1. Orquestrac com Docker Swarm

O Swarm gerencia o ciclo de vida dos containers. Sua principal func nesta arquitetura é o Self-Healing de Serviços.

  • Estado Desejado: Se configuramos 3 réplicas, o Swarm monitora constantemente. Se uma cai, ele reinstancia em outro nó.
  • Rebalanceamento: Por design, o Swarm não move containers de volta automaticamente quando um nó recupera (para evitar instabilidade). O comando --force update é usado manualmente pelo administrador para redistribuir a carga.

2. Armazenamento com GlusterFS

O GlusterFS elimina o ponto único de falha do storage tradicional (NAS/SAN).

  • Replicac Tripla: Cada bit de dado escrito no volume /docker é replicado síncronamente em todos os 3 nós.
  • Quorum: O sistema exige que a maioria dos nós (50% + 1) esteja online para aceitar escritas. Com 3 nós, o quorum é 2. Se sobrar apenas 1 nó, o volume entra em "modo de segurança" para evitar o Split-Brain.

3. Matriz de Operac (Auto vs. Manual)

Recurso / Cenário Tipo Comportamento Esperado
Detecc de Falha de Nó AUTOMÁTICO Swarm detecta e reinicia containers em nós saudáveis em segundos.
Sincronismo de Dados AUTOMÁTICO Escrita síncrona em todos os Bricks ativos via rede.
Self-Healing (GlusterFS) AUTOMÁTICO Ao voltar, o nó sincroniza dados defasados automaticamente.
Rebalanceamento de Carga MANUAL Redistribuic de containers após recuperac do cluster (Decisão Técnica).
Expansão de Disco MANUAL Ajuste de LVM e Bricks para aumentar a capacidade do volume.
